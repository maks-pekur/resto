# ADR 0003: Drizzle ORM on PostgreSQL

- **Status:** accepted
- **Date:** 2026-04-30
- **Deciders:** Resto core team

## Context

We need a type-safe data access layer over PostgreSQL that supports:

- Multi-tenancy with row-level filtering and Postgres RLS.
- Migrations as first-class artifacts, reviewed in PRs, runnable in CI/CD.
- Predictable query plans (we will tune indexes; the layer must not hide
  the actual SQL).
- Performance under load on the ordering hot path.

## Decision

Use **Drizzle ORM** with **PostgreSQL 16** as the persistence layer for
all stateful contexts. All schema and migrations live in `packages/db`;
no other package or app issues raw SQL.

## Alternatives considered

- **Prisma.** Strongest argument: mature, generates excellent typed
  clients, broad community. Rejected: query engine binary at runtime
  adds cold-start cost and operational surprise; query generation is
  opaque (harder to tune); migration story has historically been less
  flexible for non-trivial cases (RLS, partial indexes, custom types);
  type generation is heavier.
- **TypeORM.** Strongest argument: classic decorator-based ORM. Rejected:
  weaker type-safety, repository pattern is leaky, slower iteration on
  the project than the modern alternatives.
- **Kysely (query builder, no ORM).** Strongest argument: pure
  type-safe SQL, zero magic. Rejected: lacks schema/migration story
  out of the box; we would assemble half of Drizzle ourselves. We do
  use Drizzle in this query-builder-first style intentionally.
- **MikroORM.** Strongest argument: solid Unit of Work / Identity Map.
  Rejected: heavier abstraction than we want; we prefer explicit query
  shapes for the hot paths.

## Consequences

### Positive

- Type-safe queries with the SQL still visible — easier to reason about
  query plans and indexes.
- No runtime engine, no codegen step in CI hot path beyond Drizzle's
  schema-to-types inference.
- Migrations are plain SQL files generated by `drizzle-kit`, reviewable
  in PRs, with full control over RLS and custom Postgres features.
- Repository helpers in `packages/db` can encode tenant-id enforcement
  generically.

### Negative

- Smaller community than Prisma; some patterns are still emerging.
- Less hand-holding for newcomers used to Prisma's developer experience.
- We own the repository layer ourselves; that is intentional but it is
  code we maintain.

### Neutral

- Schema lives in TypeScript (`packages/db/src/schema/`) rather than a
  DSL file.

## Implementation notes

- Tenant isolation is enforced two ways:
  1. Repository helpers always inject `tenant_id` from AsyncLocalStorage.
  2. Postgres RLS policies as defense-in-depth (see ADR 0006).
- Drizzle Kit migrations are forward-only; rollbacks are implemented as
  paired forward migrations.
- Hot paths (ordering, catalog read) get explicit query shapes with
  index hints; other paths can use the higher-level relation API.
